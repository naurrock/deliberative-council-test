The system is built on three core dependencies:

- **litellm** ŌĆö Unified LLM API wrapper supporting 100+ providers through a single `completion()` interface
- **pydantic** ŌĆö Runtime-validated data models for every data structure that crosses a module boundary
- **YAML** ŌĆö Human-readable configuration with environment variable expansion

All modules are async-first. The debate and research phases use `asyncio.gather` for concurrent LLM calls, bounded by semaphores to respect rate limits.

## 2. Module Reference

### 2.1 `types.py` ŌĆö Core Type Definitions

Every data structure in the system is defined as a Pydantic `BaseModel` in this module. This provides runtime validation, JSON serialization via `model_dump_json()`, and deserialization via `model_validate_json()`. No module defines its own ad-hoc data classes ŌĆö everything flows through these types.

#### Enums

| Enum | Values | Purpose |
|------|--------|---------|
| `Complexity` | `TRIVIAL`, `MODERATE`, `COMPLEX`, `DEEP` | Question difficulty ŌĆö determines pipeline depth, debate rounds, and token budget |
| `ModelTier` | `PREMIUM`, `MID`, `CHEAP` | Model capability tier ŌĆö maps roles to cost-appropriate models |
| `EpistemicTag` | `SOURCED`, `INFERRED`, `JUDGMENT` | Epistemic status of research findings ŌĆö `SOURCED` has a URL, `INFERRED` synthesizes multiple sources, `JUDGMENT` is the agent's own assessment |
| `ConsensusLevel` | `STRONG`, `MODERATE`, `CONTESTED` | Agreement level on a key point in the final report |
| `ConvergenceReason` | 7 values | Why the debate converged or was terminated (round limit, NLI convergence, budget exhaustion, futility, etc.) |
| `ResearchMode` | `STRICT`, `AUGMENTED` | Strict mode excludes `JUDGMENT`-tagged findings from debate context; augmented includes all tags |

#### Key Data Models

**`RoleSpec`** ŌĆö A debate role generated by the Scout. Contains `name`, `perspective`, `expertise`, `suggested_model` (a family hint like `"deepseek"` or `"anthropic"`), `system_prompt` (the full prompt generated by Scout), `is_research` flag, and optional `research_subquestion`. Research roles have `is_research=True` and a non-null `research_subquestion`.

**`MissionBrief`** ŌĆö The complete pipeline configuration produced by the Scout phase. Fields include `question` (possibly rephrased), `complexity`, `domain_tags`, `is_likely_solvable`, `suggested_roles`, `research_needed`, `research_subquestions`, `debate_rounds` (0ŌĆō5), `token_budget` (for Research+Debate+Synthesis only ŌĆö Scout is tracked separately via `scout_tokens_used`), `scout_reasoning`, and `verification_notes`.

**`EvidenceReport`** ŌĆö Output of a single research agent. Contains `agent_id`, `sub_question`, `searches_performed` (list of `{query, top_results}` dicts), `key_findings` (list of `ResearchFinding`), `gaps` (what could not be found), and `tokens_used`.

**`Position`** ŌĆö An agent's position in a single debate round. Contains `agent_id`, `role_name`, `argument`, `supporting_evidence`, `self_confidence` (0.0ŌĆō1.0), `metacognitive_notes` (answer to "what would change your position?"), and `position_stability` (NLI agreement with own previous position, `None` for round 0).

**`Critique`** ŌĆö An agent's critique of another's position. Contains `from_agent`, `to_agent`, `points_of_agreement`, `points_of_disagreement`, `counter_arguments`, and `new_evidence`.

**`BehavioralSignals`** ŌĆö Per-agent metrics after a round. Includes `position_stability`, `evidence_novelty` (fraction of evidence that is new), `majority_dependence` (how much position shifted toward consensus without new evidence), and `novelty_injection_triggered`.

**`RoundResult`** ŌĆö Results of a single debate round. Contains `round_number`, `positions`, `critiques`, `agreement_matrix` (pairwise NLI scores), `nli_tier2_invoked` (whether Tier 2 was needed per pair), and `behavioral_signals`.

**`DebateState`** ŌĆö Complete state of an ongoing debate. Contains `mission_brief`, `evidence_reports`, `rounds` (list of `RoundResult`), `communication_graph`, `futility_flags`, `is_resolved`, and `resolution_reason`.

**`AgreementAnalysis`** ŌĆö Structured output from NLI Tier 2. Contains `substantively_agree`, `agreement_score`, `points_of_agreement`, `points_of_disagreement`, `is_fundamental_disagreement`, and `summary`.

**`FinalReport`** ŌĆö The final output. Contains `question`, `complexity`, `rounds_completed`, `convergence_score`, `answer` (markdown), `key_points` (list of `KeyPoint` with `ConsensusLevel`), `dissenting_views`, `research_sources`, `pipeline_trace`, `futility_notes`, and `raw_markdown`.

**`ModelInfo`** ŌĆö Runtime model metadata. Extends the config-level `ModelConfig` with `is_available` (mutable) and provider routing fields: `provider`, `api_base`, `api_key`, `conserve`, `geo_blocked`, `canonical_id`, `rpm`, `daily_quota`, `regenerates`.

**`ProviderInfo`** ŌĆö Provider-level metadata from the `providers:` section of `providers.yaml`. Contains `name`, `priority` (lower = preferred), `env_key`, `regenerates`, `daily_quota`, `rpm`, `geo_blocked`, `conserve`, `api_base_template`, and `notes`.

**`HealthCheckResult`** ŌĆö Result of pinging a model. Contains `model_id`, `is_healthy`, `latency_ms`, and `error`.

### 2.2 `config.py` ŌĆö Configuration Loading and Validation

This module handles loading, parsing, and validating the YAML configuration. The top-level class is `CouncilConfig`, which aggregates all sub-configs.

#### Configuration Classes

**`ModelConfig`** ŌĆö Per-model configuration. Mirrors `ModelInfo` but with an `enabled` field and without the mutable `is_available`. The `api_base` and `api_key` fields may contain `${ENV_VAR}` references that are expanded at load time.

**`FallbackChain`** ŌĆö Maps a `role_type` (one of `"research"`, `"debater"`, `"synthesizer"`) to an ordered list of model IDs.

**`NLIConfig`** ŌĆö Two-tier NLI configuration. Key thresholds:
- `tier1_uncertain_low` (default 0.25): Below this, Tier 1 is confident about disagreement
- `tier1_uncertain_high` (default 0.65): Above this, Tier 1 is confident about agreement
- `convergence_threshold` (default 0.75): NLI agreement threshold for convergence detection
- `convergence_rounds` (default 2): Consecutive rounds above threshold needed
- `position_stability_threshold` (default 0.80): Triggers novelty injection
- `graceful_degradation` (default True): Falls back to cheap LLM if DeBERTa fails to load

**`BudgetConfig`** ŌĆö Token budget configuration. The default budget is 500,000 tokens, split 30% research / 50% debate / 20% synthesis. Shares are validated to be positive.

**`DebateConfig`** ŌĆö Debate phase settings. `default_rounds` maps complexity levels to round counts (trivial=0, moderate=1, complex=2, deep=3). `max_concurrent_agents` (default 3) controls the concurrency semaphore.

**`ResearchConfig`** ŌĆö Research phase settings. `mode` (`STRICT` or `AUGMENTED`), `jina_search_url` and `jina_extract_url` for Jina.ai endpoints, and `max_search_results` (default 5).

#### Loading Pipeline

`load_config(config_path, **overrides)` implements a four-level priority system:

1. **Explicit keyword arguments** (from CLI flags) ŌĆö highest priority
2. **YAML file values** ŌĆö loaded from `config_path` or auto-discovered
3. **Auto-discovered config** ŌĆö `./config/providers.yaml` then `~/.config/deliberative-council/providers.yaml`
4. **Built-in defaults** ŌĆö `DEFAULT_MODELS` and `DEFAULT_FALLBACK_CHAINS`

If no config file is found, the built-in defaults (6 free-tier models across Cloudflare and OpenRouter) are used.

### 2.3 `models.py` ŌĆö Model Registry and LLM Client

This is the largest and most complex module, handling model selection, failure tracking, fallback resolution, and the litellm wrapper.

#### `TokenBudget`

A simple dataclass tracking token consumption: `total`, `used`, `remaining` (property), `is_exhausted` (property), `consume()`, and `can_allocate()`. The engine creates separate `TokenBudget` instances for each phase to prevent double-counting.

#### `FailureSeverity` and `FailureTracker`

The two-strikes escalation system:

- **Strike 1 (TRANSIENT)**: 60-second cooldown. Any failure ŌĆö 429, timeout, 403, network error ŌĆö triggers this. The model might just be rate-limited and will recover.
- **Strike 2 (DAILY)**: Cooldown until midnight UTC. If the same model fails again while its TRANSIENT record is still active, it escalates to DAILY. This indicates RPD quota exhaustion, a revoked key, or other persistent issue.

There is no "permanent" severity. Models known to be unreachable (geo-blocked) are filtered at selection time and never reach the tracker. Runtime-discovered permanent failures simply re-escalate to DAILY each day, costing at most one attempt per day.

`FailureTracker` maintains `_model_failures` (a `dict[str, list[FailureRecord]]`) and `_strike_count` (a `dict[str, int]`). On `record_success()`, the strike count resets to 0 and transient records are cleared (but DAILY records persist until midnight). The `get_failure_summary()` method returns a diagnostic dict of all active failures.

#### `ModelRegistry`

The registry maintains the model pool and implements selection algorithms. Key data structures:

- `_models: dict[str, ModelInfo]` ŌĆö All registered models by `model_id`
- `_fallback_chains: dict[str, list[str]]` ŌĆö Fallback chains by role type
- `_rr_counters: dict[str, int]` ŌĆö Round-robin counters keyed by `"{tier}:{family}"`
- `_canonical_index: dict[str, list[str]]` ŌĆö Canonical ID ŌåÆ list of model_ids sharing that ID
- `_providers: dict` ŌĆö Provider metadata for priority-aware fallback

The `select_model_for_role()` method implements the core selection algorithm (detailed in Section 4 below). Key methods for querying the pool: `available_models()` (excludes geo-blocked), `selectable_models()` (excludes geo-blocked and daily-exhausted), `models_by_family()`, `models_by_tier()`, `models_by_provider()`, `models_by_canonical()`.

#### `LLMClient`

The litellm wrapper that adds budget tracking, failure tracking, retry logic, and fallback resolution. Detailed in Section 5 below.

### 2.4 `scout.py` ŌĆö Scout Phase

The Scout phase uses a two-agent architecture to classify the question and produce a `MissionBrief`.

#### `run_scout(question, client, registry, tools, config)`

1. **Model selection**: Selects a CHEAP-tier model for the scout agent (`_select_scout_model`) and a MID-tier model for the verifier agent (`_select_verifier_model`). If no CHEAP models are available, falls back to the first available model; if no MID models, falls back to PREMIUM.

2. **Preliminary search**: If a `ToolRegistry` is provided, runs a quick web search via `_preliminary_search()` to give the scout context. Failures are logged and silently ignored ŌĆö the scout proceeds without search results.

3. **Scout agent**: Sends the question (plus search context, if any) to the CHEAP model with the `SCOUT_SYSTEM_PROMPT`, which instructs it to return a JSON object with complexity classification, domain tags, suggested roles, research subquestions, debate rounds, and token budget. Temperature is 0.4 for reasonable consistency.

4. **Verifier agent**: Sends the question and the scout's draft to a MID-tier model with `VERIFIER_SYSTEM_PROMPT`. The verifier catches common scout errors: misclassifying trick questions as trivial, under-budgeting, too few roles, and missing perspectives. Temperature is 0.3 for higher accuracy.

5. **Parsing**: `_parse_mission_brief()` handles common LLM output issues (markdown code fences, invalid complexity strings, missing fields). If the LLM output can't be parsed, a safe MODERATE-complexity default is returned. If there are fewer than 2 roles for a non-trivial question, `_add_default_roles()` fills in from a preset list (Analytical Expert, Skeptical Reviewer, Pragmatic Realist).

6. **Overrides**: After parsing, `complexity_override` and `budget_override` from the config are applied, and `scout_tokens_used` is recorded on the brief.

The Scout phase has its own separate `TokenBudget(total=50_000)` that is not deducted from the main pipeline budget. This prevents the Scout from starving later phases.

### 2.5 `research.py` ŌĆö Research Phase

Deploys parallel search-and-ingest agents using Jina.ai for web content. Each agent investigates a sub-question and produces an `EvidenceReport` with epistemic tags and multi-source URL support.

#### `run_research(brief, client, registry, tools, budget, config, research_mode)`

1. **Early exit**: If `brief.research_needed` is `False` or `brief.research_subquestions` is empty, returns an empty list immediately.

2. **Budget allocation**: The budget passed in is already the research sub-budget (engine.py applies `research_share` before calling). The total is divided evenly across sub-questions. **Important**: `research_share` is NOT applied again here ŌĆö that would double-reduce the budget.

3. **Role creation**: `_create_research_roles()` creates one `RoleSpec` per sub-question with `is_research=True`, `suggested_model=""` (letting the registry pick from CHEAP tier), and the sub-question as the `research_subquestion`.

4. **Concurrent execution**: Agents run concurrently via `asyncio.gather`, bounded by a `Semaphore(research_cfg.max_concurrent_agents)`. Each agent that throws an exception produces a minimal `EvidenceReport` with the error in `gaps`.

#### `_run_research_agent(role, client, registry, tools, budget, config, research_mode)`

Each agent follows a three-step process:

1. **Search**: Calls `tools.execute("web_search", query=sub_question)` via the Jina.ai search endpoint.
2. **Extract**: For the top 3 search results, calls `tools.execute("extract_content", url=...)` to get clean text (capped at 5000 chars each).
3. **Synthesize**: Sends the search context and raw snippets to a CHEAP-tier LLM with the `RESEARCH_AGENT_PROMPT`, which instructs it to tag findings with epistemic status and return structured JSON.

#### `_parse_research_findings(response_text, search_results, research_mode)`

Parses the LLM's JSON output into `ResearchFinding` objects. In **strict mode**:
- `SOURCED` findings without a URL are downgraded to `INFERRED`
- `JUDGMENT`-tagged findings are filtered out entirely
- This ensures the debate context only contains verifiable information

In **augmented mode**, all tags are allowed but clearly labeled.

### 2.6 `debate.py` ŌĆö Debate Phase

Orchestrates structured debate rounds with NLI agreement tracking, position stability, novelty injection, and futility detection.

#### `run_debate(brief, evidence, client, registry, budget, prior_state, config, ...)`

1. **State initialization**: Creates or resumes a `DebateState`. Only non-research roles participate in the debate.

2. **Communication graph**: Built via `build_communication_graph()` (currently full graph only).

3. **Model assignment**: `_assign_models_to_agents()` selects models for each debater, passing `already_assigned_families` to maximize family diversity. Raises `RuntimeError` if no model is available for any debater.

4. **Round loop**: Runs from `starting_round` to `max_rounds`, checking budget before each round. Each round:
   - Assigns devil's advocate roles (currently strategy `"none"`)
   - Runs `_run_debate_round()` (detailed below)
   - Updates stability history from `behavioral_signals`
   - Checks convergence via `check_convergence()` ŌĆö stops if converged
   - Checks futility via `check_futility_heuristic()` ŌĆö if ambiguous (confidence Ōēź 0.3), escalates to `check_futility_llm()`

5. **Termination**: If all rounds are exhausted without resolution, sets `resolution_reason` to `ROUND_LIMIT_REACHED`.

#### `_run_debate_round(round_num, brief, evidence, state, client, registry, budget, ...)`

A single round has five steps:

1. **Generate positions** (concurrent): Each agent generates a position using the `DEBATE_POSITION_PROMPT`, which includes their role, the question, research context, and prior rounds. If `should_inject_novelty()` returns True, the `NOVELTY_INJECTION_PROMPT` is appended, forcing the agent to argue from a new angle. Positions are parsed from JSON via `_parse_position()`, which falls back to using the raw text as the argument if JSON parsing fails.

2. **Compute position stability**: For rounds > 0, each agent's current position is compared to their previous position via `compute_position_stability()` (DeBERTa NLI). The result is stored on the `Position` object.

3. **Generate critiques** (sequential): Each agent critiques each visible agent's position using `DEBATE_CRITIQUE_PROMPT`. Critiques are generated one at a time (not concurrently) to avoid overwhelming the API.

4. **Compute agreement matrix** (NLI two-tier): For each unique pair (i < j), `compute_agreement()` returns a score and whether Tier 2 was invoked. The matrix is symmetric. On failure, a default score of 0.5 is assigned.

5. **Compute behavioral signals**: For each agent, records `position_stability` and whether `novelty_injection_triggered`. The `evidence_novelty` and `majority_dependence` fields are not yet computed (left at defaults).

### 2.7 `synthesis.py` ŌĆö Synthesis Phase

Uses an impartial model from a **different family** than the debaters to produce the final `FinalReport`.

#### `run_synthesis(brief, evidence, debate_state, client, registry, config)`

1. **Select synthesizer**: `_select_synthesizer()` picks a model from a family NOT used by any debater, preferring PREMIUM tier. If all premium families are taken, it still uses premium for quality. Falls back to MID-tier if no premium models exist. The family exclusion is determined by examining `registry.get_usage()` ŌĆö any model that was used during the debate is considered a "debater family".

2. **Build summaries**: `_build_debate_summary()` produces a structured summary of all rounds (positions, agreement scores, key disagreements, futility flags). `_build_evidence_summary()` lists research findings with epistemic tags and source URLs.

3. **Generate synthesis**: The `SYNTHESIS_PROMPT` instructs the model to weigh arguments by evidence quality, preserve minority positions, and assign consensus levels. Temperature is 0.3 for accuracy.

4. **Parse synthesis**: `_parse_synthesis()` extracts the answer, key points with consensus levels, dissenting views, and research sources. If JSON parsing fails, `_fallback_report()` creates a minimal report from the last round's positions.

5. **Build pipeline trace**: `_build_pipeline_trace()` maps model usage to roles using `_infer_role()` (which checks fallback chain membership), computes per-phase token totals, and adds the synthesizer's own usage.

6. **Generate markdown**: `_generate_markdown()` produces the complete markdown document stored in `report.raw_markdown`.

### 2.8 `feasibility.py` ŌĆö Futility Detection

Two-layer approach to detect when debate is making no progress.

#### `check_futility_heuristic(debate_state)`

No LLM cost. Checks three signals:

1. **All agents stuck**: If every agent in the latest round has `position_stability > 0.85` AND there have been 3+ rounds, returns `is_futile=True` with confidence 0.85.

2. **Agreement matrix frozen**: If the average agreement score changes by less than 0.02 between the last two rounds AND there have been 3+ rounds, returns `is_futile=True` with confidence 0.75.

3. **Majority stuck**: If Ōēź75% of agents have `position_stability Ōēź 0.80` after 2+ rounds, returns `is_futile=False` with confidence 0.5 ŌĆö suggesting LLM escalation rather than declaring futility outright.

The heuristic is deliberately conservative: it only declares futility directly when the evidence is overwhelming. Ambiguous cases return moderate confidence to trigger the LLM layer.

#### `check_futility_llm(debate_state, client, model_id)`

Escalates to an LLM when heuristics are ambiguous (confidence in 0.3ŌĆō0.7 range). Sends a summary of the last 3 rounds (positions, confidence scores, agreement averages) and asks the LLM to determine if further rounds would help. The LLM responds with `{is_futile, reason, confidence}`. On JSON parse failure, returns `is_futile=False` to avoid terminating a potentially productive debate.

### 2.9 `nli.py` ŌĆö Two-Tier NLI Agreement System

Implements agreement measurement, convergence detection, position stability, and novelty injection.

#### Tier 1: DeBERTa (`tier1_agreement`)

Uses `cross-encoder/nli-deberta-v3-large` loaded lazily on first use. The model is forced to CPU. For each pair of texts:

1. Both texts are chunked to fit within DeBERTa's 512-token limit (500 tokens per chunk, preserving sentence boundaries via `_chunk_text()`)
2. Each chunk pair is classified as contradiction/neutral/entailment
3. The entailment probability serves as the agreement score
4. For multi-chunk inputs, scores are averaged across all chunk pairs

If DeBERTa fails to load (e.g., no `transformers` or `torch` installed), `_heuristic_agreement()` provides a Jaccard-based fallback that returns scores in the 0.35ŌĆō0.60 range (the "uncertain zone"), ensuring Tier 2 will be invoked. This is the `graceful_degradation` behavior.

#### Tier 2: LLM (`tier2_agreement`)

Invoked only when Tier 1 returns a score in the uncertain zone (between `tier1_uncertain_low` and `tier1_uncertain_high`). Sends both positions to the configured `tier2_model` and asks for a structured `AgreementAnalysis` with `substantively_agree`, `agreement_score`, `points_of_agreement`, `points_of_disagreement`, and `is_fundamental_disagreement`. On parse failure, returns a default uncertain analysis with `agreement_score=0.5`.

#### `compute_agreement(position_a, position_b, client, config)`

The combined two-tier function. Always runs Tier 1 first. If the score falls in the confident zones (Ōēż0.25 or Ōēź0.65), returns the Tier 1 score directly. Otherwise, escalates to Tier 2 and returns that score. Returns `(agreement_score, tier2_was_invoked)`.

#### `check_convergence(debate_state, config, budget_remaining)`

Hybrid convergence check with priority ordering:

1. **Budget exhaustion** (hard ceiling): If `budget_remaining Ōēż 0`, converges with `BUDGET_EXHAUSTED`
2. **Round limit** (soft ceiling): If rounds completed Ōēź `debate_rounds`, converges ŌĆö but if average agreement < 0.5, returns `ROUND_LIMIT_BUT_DISAGREEMENT` (the debate continues)
3. **NLI convergence** (adaptive): If the last `convergence_rounds` consecutive rounds all have average agreement Ōēź `convergence_threshold`, converges with `NLI_TIER1_CONVERGENCE`

#### `compute_position_stability(current, previous, config)`

Compares an agent's current position to their previous position using Tier 1 NLI. Returns `None` for the first round. Higher scores mean more stability (less change).

#### `should_inject_novelty(stability_history, config)`

Triggers when `position_stability Ōēź position_stability_threshold` (default 0.80) for `position_stability_rounds` (default 2) consecutive rounds. When triggered, the `NOVELTY_INJECTION_PROMPT` is appended to the agent's position prompt, forcing them to argue from a new angle, challenge their own assumptions, and introduce new evidence.

### 2.10 `engine.py` ŌĆö Pipeline Orchestrator

The main entry point that orchestrates the full pipeline with checkpoint/resume support.

#### `run_council(question, config, run_id)`

1. **Initialization**: Creates a `ModelRegistry` from config, a `ToolRegistry` (with Jina client), and a `TokenBudget`. The `LLMClient` is created with `budget=None` to detach the global budget ŌĆö each phase tracks its own sub-budget and reports consumption back manually.

2. **Checkpoint check**: If `config.resume` is True, checks for an existing checkpoint and skips completed phases.

3. **Phase 1 (Scout)**: Runs `run_scout()`. Saves the brief to the checkpoint. If complexity is TRIVIAL, skips directly to synthesis with an empty `DebateState`.

4. **Phase 2 (Research)**: If `brief.research_needed`, creates a research sub-budget (`total ├Ś research_share`) and runs `run_research()`. The used tokens are consumed from the main budget. Saves evidence to checkpoint.

5. **Phase 3 (Debate)**: Creates a debate sub-budget (`total ├Ś debate_share`) and runs `run_debate()`. Supports resuming from a prior `DebateState`. Saves debate state to checkpoint.

6. **Phase 4 (Synthesis)**: Runs `run_synthesis()` and updates the pipeline trace with scout tokens.

7. **Cleanup**: Closes the Jina HTTP session in a `finally` block.

#### `CheckpointManager`

SQLite-based crash recovery. See Section 7 for full details.

### 2.12 `graph.py` ŌĆö Communication Graph

Currently implements only the **full graph** strategy, where every agent sees every other agent's positions. The `_build_full_graph()` function creates a dict mapping each agent to a list of all other agents.

The `validate_graph()` function checks three invariants: every agent appears as a key, no agent appears in its own visibility list, and all referenced agents exist in the agent list. Returns `True`/`False` with warnings logged for each violation.

The `sparse` strategy is an extension stub that raises `NotImplementedError`. The rationale for full-only: with 2ŌĆō4 debaters, the token savings from sparse communication are negligible, while the implementation complexity and debugging difficulty are substantial.

---

### 2.13 `context.py` ŌĆö Context Window Management

Builds the context string for each agent's debate turn. Currently implements only the **full** strategy.

`build_agent_context(agent_id, round_num, graph, all_positions, research_evidence, strategy, max_context_chars)` produces a formatted string with two sections:

1. **Research Evidence**: Lists all `EvidenceReport` findings with epistemic tags, relevance scores, and source URLs.
2. **Prior Debate Rounds**: For each prior round, shows the positions of all visible agents (including the agent's own prior positions). Visibility is determined by the communication graph.

If the context exceeds `max_context_chars` (default 80,000), `_compress_context()` truncates from the front (keeping the most recent content) with a truncation note.

The `progressive` strategy (gradually reveal information) is an extension stub.

---

### 2.14 `devil_advocate.py` ŌĆö Devil's Advocate Assignment

Currently implements only the `none` strategy, which assigns no devil's advocate to any agent. The `assign_devils_advocate()` function returns a dict of `{agent_id: DevilAdvocateAssignment()}` with all fields set to `None`.

Three extension stubs are defined but not implemented: `rotate` (cycle through agents), `weakest` (assign to the least confident agent), and `random`. Each raises `NotImplementedError`.

The `DevilAdvocateAssignment` dataclass contains `devil_agent_id`, `original_role`, and `devil_prompt_addition` ŌĆö all optional, all `None` in the current implementation.

## 3. Configuration System

### 3.1 `providers.yaml` Structure

The primary configuration file is `config/providers.yaml`. It has three top-level sections:

```yaml
providers:    # Provider metadata (priority, quotas, geo-blocking)
models:       # Model registry (one entry per model variant)
fallback_chains:  # Ordered fallback lists per role type
```

### 3.2 Provider Section Parsing

The `providers:` section is parsed into `ProviderInfo` objects. Each key is the provider name (e.g., `cloudflare`, `openrouter`), and the value is a dict of metadata. The parsing process:

1. The provider name is injected as the `name` field
2. `api_base_template` is expanded via `_expand_env_vars()` (e.g., `{CLOUDFLARE_ACCOUNT_ID}` ŌåÆ actual account ID)
3. The dict is passed to `ProviderInfo(**info)` for validation
4. The result is stored in `CouncilConfig.providers` as `dict[str, ProviderInfo]`

Provider metadata is used by the registry for:
- **Priority ordering** in cross-provider fallback (lower priority number = preferred)
- **Conserve awareness** (non-regenerating credits are used sparingly)
- **Geo-blocking** (providers blocked from the server's region)
- **Quota awareness** (daily quota and regeneration status)

### 3.3 Model Section Parsing

The `models:` section is parsed into `ModelConfig` objects. Key processing steps:

1. **Environment variable expansion**: `api_base` and `api_key` fields containing `${ENV_VAR}` patterns are expanded via `_expand_env_vars()`. For example, `${CLOUDFLARE_API_KEY}` is replaced with the value of the `CLOUDFLARE_API_KEY` environment variable. If the variable is not set, the pattern is left as-is.

2. **Extra key stripping**: After env var expansion, each model dict is filtered to only include keys that are valid `ModelConfig` fields. This is necessary because `providers.yaml` may include extra metadata (like `env_account_id` at the provider level) that isn't a `ModelConfig` field. The filter uses `set(ModelConfig.model_fields.keys())`.

3. **Fallback chain normalization**: The `fallback_chains:` section in the YAML uses a dict format (`{role_type: [model_ids]}`), but `CouncilConfig` expects a list of `FallbackChain(role_type=..., chain=[...])`. The normalization converts:
   ```yaml
   fallback_chains:
     research: [model_a, model_b]
     debater: [model_c, model_d]
   ```
   into:
   ```python
   [
     {"role_type": "research", "chain": ["model_a", "model_b"]},
     {"role_type": "debater", "chain": ["model_c", "model_d"]},
   ]
   ```

### 3.4 Auto-Discovery

If no explicit config path is provided, `_auto_find_config()` searches in order:

1. `./config/providers.yaml` (relative to CWD)
2. `~/.config/deliberative-council/providers.yaml`

If neither file exists, the built-in `DEFAULT_MODELS` and `DEFAULT_FALLBACK_CHAINS` are used. These contain 6 free-tier models (3 Cloudflare, 3 OpenRouter) and 3 basic fallback chains.

### 3.5 Override Priority

The full priority chain (highest to lowest):

1. CLI keyword arguments (passed as `**overrides` to `load_config()`)
2. YAML file values
3. Auto-discovered config file values
4. Built-in defaults

Only non-`None` override values are applied. This means CLI flags like `--budget 200000` override the YAML's `budget.default_budget`, but omitting `--budget` preserves the YAML value.

## 4. Model Registry Internals

### 4.1 Selection Algorithm

`select_model_for_role(role, *, family, exclude_families, local_only, api_only, model_override, already_assigned_families)` implements a multi-step selection:

1. **Explicit override**: If `model_override` is provided and the model is available (not geo-blocked), use it immediately.

2. **Target tier**: Determined by `_tier_for_role()` ŌĆö research roles get CHEAP, debate roles get MID. (Synthesizer selection is handled separately in `synthesis.py`.)

3. **Candidate filtering**: Start with `available_models()` (excludes geo-blocked), then apply `local_only`, `api_only`, `family`, and `exclude_families` constraints.

4. **Family diversity**: If `already_assigned_families` is provided and no target family is specified, prefer models from unassigned families. If all families are assigned, sort by least-used family.

5. **Tier matching**: Filter candidates to the target tier. If no matches, try the adjacent tier (PREMIUMŌåÆMID, MIDŌåÆCHEAP, CHEAPŌåÆany).

6. **Round-robin selection**: Within the filtered pool, `_rr_select()` cycles through candidates.

### 4.2 Round-Robin Selection (`_rr_select`)

The round-robin key format is `"{tier}:{family}"` (e.g., `"mid:llama"`, `"cheap:__all__"`). This ensures that different selection contexts rotate independently.

Before round-robin, candidates are sorted by health:

```python
sort_key = (daily_exhausted, cooling_down, conserved)
# Lower is better: (False, False, False) > (True, True, True)
```

The selection then proceeds through a cascade:

1. If ALL candidates are cooling down, pick the least-bad option (minimize daily exhaustion, then failure count)
2. Otherwise, filter out daily-exhausted and cooling-down models
3. If no healthy models remain, relax to cooling-down-but-not-daily
4. If still empty, include daily-exhausted as last resort
5. Filter out conserved models (if non-conserved alternatives exist)
6. Round-robin within the remaining pool: `idx = rr_counter[key] % len(pool)`

### 4.3 Cross-Provider Fallback (`_cross_provider_fallback`)

When a model fails and has a `canonical_id`, the registry looks for the same underlying model on a different provider. The process:

1. Look up `canonical_id` in `_canonical_index` to get all models sharing that ID
2. Filter out the failed model, geo-blocked models, and models that are daily-exhausted or cooling down
3. Sort remaining alternatives by provider priority (lower = preferred)
4. Prefer non-conserved providers

For example, if `openai/@cf/meta/llama-3.3-70b-instruct-fp8-fast` (canonical_id: `llama-3.3-70b-instruct`) fails, the registry will try `openrouter/meta-llama/llama-3.3-70b-instruct:free` (same canonical_id, different provider).

### 4.4 Fallback Resolution (`resolve_fallback`)

When a model fails, `resolve_fallback(model_id, role_type)` tries three strategies in order:

| Priority | Strategy | Description |
|----------|----------|-------------|
| 1 | Cross-provider | Same model (canonical_id) on a different provider ŌĆö preserves reasoning capability |
| 2 | Explicit chain | Walk the configured `fallback_chains[role_type]` list, skipping unavailable, geo-blocked, daily-exhausted, and cooling-down models |
| 3 | Same-tier round-robin | Any available model at the same tier that isn't the failed model and isn't daily-exhausted or cooling down |

This three-strategy approach ensures that the system prefers the highest-value fallback first (same reasoning, different route) before falling back to different models entirely.

## 5. LLM Client Error Handling

### 5.1 `LLMClient.complete()` Flow

```python
async def complete(model_id, messages, max_tokens, temperature, ...) -> tuple[str, int]
```

The method implements a retry-with-fallback pattern:

```
Attempt 0 ŌöĆŌöĆŌ¢║ Success ŌöĆŌöĆŌ¢║ Return (content, tokens)
     Ōöé
     Failure ŌöĆŌöĆŌ¢║ Record failure (strike 1 = TRANSIENT)
     Ōöé
     Ōö£ŌöĆ If DAILY (strike 2+) ŌöĆŌöĆŌ¢║ resolve_fallback() ŌöĆŌöĆŌ¢║ Recurse with fallback model
     Ōöé
     ŌööŌöĆ If TRANSIENT (strike 1) ŌöĆŌöĆŌ¢║ Wait 1s ŌöĆŌöĆŌ¢║ Attempt 1 ŌöĆŌöĆŌ¢║ Success ŌöĆŌöĆŌ¢║ Return
                                                        Ōöé
                                                        Failure ŌöĆŌöĆŌ¢║ Record (strike 2 = DAILY)
                                                        Ōöé
                                                        Ōö£ŌöĆ If DAILY ŌöĆŌöĆŌ¢║ resolve_fallback() ŌöĆŌöĆŌ¢║ Recurse
                                                        Ōöé
                                                        ŌööŌöĆ Wait 4s ŌöĆŌöĆŌ¢║ Attempt 2 ŌöĆŌöĆŌ¢║ Success ŌöĆŌöĆŌ¢║ Return
                                                                            Ōöé
                                                                            Failure ŌöĆŌöĆŌ¢║ resolve_fallback() ŌöĆŌöĆŌ¢║ Recurse
                                                                                        Ōöé
                                                                                        No fallback ŌöĆŌöĆŌ¢║ Raise RuntimeError
```

### 5.2 Two-Strikes Escalation

Each failure is recorded via `FailureTracker.record_failure()`. The escalation logic:

- **Strike 1**: `TRANSIENT` severity, 60-second cooldown. The model might be rate-limited and will recover. The retry logic waits 1 second before the next attempt.
- **Strike 2+**: `DAILY` severity, cooldown until midnight UTC. This indicates RPD quota exhaustion or a persistent issue. Retries are skipped immediately ŌĆö the system goes straight to fallback.

A successful call (`record_success()`) resets the strike count to 0 and clears TRANSIENT records. DAILY records persist until midnight but won't cause further issues since the strike count is reset.

### 5.3 DAILY Exhaustion ŌåÆ Immediate Fallback

When a failure escalates to DAILY, the remaining retries for that model are skipped entirely. There's no point retrying a model that's hit its daily quota. The system immediately calls `resolve_fallback()` and recurses with the fallback model.

### 5.4 TRANSIENT ŌåÆ Retry with Backoff

For TRANSIENT failures, the client retries with exponential backoff (1s, 4s for the two retries). After all retries are exhausted, it tries `resolve_fallback()`. If a fallback is found, the call is recursively made to the fallback model with the same parameters.

### 5.5 Reasoning Model Support

After receiving a response, the client checks if `content` is empty. If so, it looks for `reasoning_content` on the response message (litellm's field for reasoning models like DeepSeek R1). If `reasoning_content` exists, it's used as the response text. This ensures compatibility with models that put their output in the reasoning field rather than the content field.

### 5.6 Budget Tracking

Before making a call, the client checks if the active budget (`budget_override` or `self.budget`) is exhausted. If so, it raises `RuntimeError("Token budget exhausted")`. After a successful call, token usage is recorded both in the active budget (`budget.consume()`) and in the registry (`registry.record_usage()`).

## 6. Cloudflare Integration

### 6.1 The OpenAI-Compatible Bypass

Cloudflare Workers AI models are accessed through litellm's `openai/` prefix with a custom `api_base` pointing to Cloudflare's OpenAI-compatible endpoint. This is a deliberate design decision to bypass litellm's native `cloudflare/` handler, which has known issues:

- The native handler uses the old `/ai/run/` endpoint, which breaks on newer model response formats
- The `/ai/v1` endpoint (OpenAI-compatible) is more reliable and follows the standard chat completions format

### 6.2 Configuration Pattern

In `providers.yaml`, Cloudflare models follow this pattern:

```yaml
- model_id: openai/@cf/meta/llama-3.3-70b-instruct-fp8-fast
  family: llama
  tier: mid
  provider: cloudflare
  api_base: "${CLOUDFLARE_API_BASE}"
  api_key: "${CLOUDFLARE_API_KEY}"
```

The `openai/` prefix tells litellm to use the OpenAI-compatible handler. The `@cf/` prefix in the model name is Cloudflare's internal model identifier. The `api_base` and `api_key` fields are expanded from environment variables at load time.

### 6.3 Environment Variable Setup

Two environment variables are required for Cloudflare:

- `CLOUDFLARE_API_KEY`: The Cloudflare API key (set directly)
- `CLOUDFLARE_API_BASE`: Expanded from `CLOUDFLARE_ACCOUNT_ID` using the provider's `api_base_template`:
  ```
  https://api.cloudflare.com/client/v4/accounts/{CLOUDFLARE_ACCOUNT_ID}/ai/v1
  ```

The `api_base_template` in the providers section uses `{CLOUDFLARE_ACCOUNT_ID}` (curly braces, not `${}` syntax), which is expanded by `_expand_env_vars()` during config loading.

### 6.4 Model Coverage

14 models are registered across 11 families on Cloudflare, spanning all three tiers:

| Tier | Families | Model Count |
|------|----------|-------------|
| CHEAP | llama, granite | 3 models |
| MID | llama, qwen, deepseek, gpt-oss, llama4, gemma, mistral, nemotron, glm, moonshot, qwq | 11 models |
| PREMIUM | gpt-oss, nemotron, moonshot | 3 models |

This gives the debate system maximum family diversity ŌĆö up to 11 different model families for debaters, each with genuinely different reasoning characteristics. The Cloudflare models are at priority 0 (highest), with daily-regenerating neuron budgets of ~10K neurons/day.

## 7. Checkpoint System

### 7.1 SQLite-Based Crash Recovery

The `CheckpointManager` class in `engine.py` provides crash recovery through SQLite checkpoints. The database is stored at `.council/checkpoints.db` (created automatically).

#### Schema

```sql
CREATE TABLE IF NOT EXISTS checkpoints (
    run_id TEXT PRIMARY KEY,
    created_at TEXT,
    brief_json TEXT,
    evidence_json TEXT,
    debate_state_json TEXT,
    report_json TEXT,
    phase TEXT
)
```

#### Save Points

The pipeline saves after each phase:

| Phase | Method | Data Saved | Phase Value |
|-------|--------|-----------|-------------|
| Scout | `save_brief()` | `MissionBrief.model_dump_json()` | `"scout"` |
| Research | `save_evidence()` | `json.dumps([e.model_dump() for e in evidence])` | `"research"` |
| Debate | `save_debate_state()` | `DebateState.model_dump_json()` | `"debate"` |

The `INSERT OR REPLACE` SQL statement ensures that re-running with the same `run_id` overwrites the previous checkpoint.

### 7.2 Resume Support

When `--resume` is passed, the engine checks for an existing checkpoint with the same `run_id`. For each phase, it checks whether the phase was already completed:

- If the checkpoint's `phase` is `"research"` or later, the `MissionBrief` is deserialized from `brief_json` via `MissionBrief.model_validate_json()`
- If the phase is `"debate"` or later, the evidence list is deserialized from `evidence_json` via `json.loads()` + `EvidenceReport(**e)`
- If the phase is `"synthesis"`, the debate state is deserialized from `debate_state_json` via `DebateState.model_validate_json()`

Each skipped phase logs a message like `"Resumed from checkpoint: scout phase already complete"`.

### 7.3 JSON Serialization

All Pydantic models support JSON serialization via `model_dump_json()` (serialization) and `model_validate_json()` (deserialization). The evidence list is an exception ŌĆö since it's a `list[EvidenceReport]`, it's serialized as `json.dumps([e.model_dump() for e in evidence])` and deserialized with `json.loads()` + individual `EvidenceReport(**e)` construction.

### 7.4 Run ID Generation

If no `run_id` is provided, one is generated as `str(uuid.uuid4())[:8]` (first 8 characters of a UUID). For resume to work, the same `run_id` must be used across runs. The CLI currently generates a new ID for each invocation, so `--resume` will only work if the previous run was in the same process or the ID was logged and manually specified.
